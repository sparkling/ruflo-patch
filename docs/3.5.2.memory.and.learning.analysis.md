# 3.5.2 Memory & Learning Systems Analysis

> **Date**: 2026-03-05 (updated)
> **Scope**: ruflo (formerly claude-flow) v3.5.2, AgentDB v3, related packages
> **Purpose**: Inventory all memory/learning systems, assess integration status, identify wiring opportunities

---

## Executive Summary

The ruflo 3.5.2 release represents a massive memory/learning overhaul integrating AgentDB v3. However, analysis reveals **8 active persistence mechanisms and 4+ learning systems running in parallel**, many overlapping, with significant features available but not wired. This report provides decision matrices to prioritize integration.

**Key Finding**: The system has more capability defined than wired. ~40% of available AgentDB v3 features are actively integrated. The remaining 60% are documented in skills/agents but not connected to the runtime.

**Critical Finding**: AgentDB's native RVF format is never materialized on disk — no `.rvf` files exist. The hybrid backend falls back to SQLite everywhere. Meanwhile, **two separate SQLite databases** (`.claude-flow/learning/patterns.db` via `better-sqlite3` and `.swarm/memory.db` via `sql.js`) store overlapping pattern/trajectory data using different engines and incompatible schemas, with no synchronization between them.

### The Two-SQLite Problem

The system runs **two separate SQLite databases** with overlapping schemas and no sync:

| Attribute | Learning DB (`.claude-flow/learning/patterns.db`) | Swarm DB (`.swarm/memory.db`) |
|-----------|---------------------------------------------------|-------------------------------|
| **Engine** | `better-sqlite3` (native C++ addon) | `sql.js` (WASM-compiled SQLite) |
| **Pattern tables** | `short_term_patterns`, `long_term_patterns` | `patterns`, `pattern_history` |
| **Trajectory tables** | `trajectories` (flat) | `trajectories` + `trajectory_steps` (with rewards) |
| **Memory entries** | None | `memory_entries` (typed: semantic, episodic, procedural, working, pattern) |
| **HNSW metadata** | `hnsw_index` (vector ID mappings) | `vector_indexes` (full config: M, efConstruction, quantization) |
| **Session tracking** | `session_state` | `sessions` (with project_path, branch, metrics) |
| **Migrations** | None | `migration_state` (resume-capable) |
| **Writer** | `learning-service.mjs` | `@claude-flow/cli` (swarm init) |
| **Synced?** | **No** | **No** |

The swarm DB has the more complete schema (10 tables, versioned patterns, typed memory entries, migration tracking), but the learning DB is the one actively written to by the HNSW learning service. They store similar data independently.

## 3. Overlap & Redundancy Analysis

### Identified Redundancies

| Overlap | System A | System B | Recommendation |
|---------|----------|----------|----------------|
| **Two SQLite DBs** | `patterns.db` (`better-sqlite3`, native) | `.swarm/memory.db` (`sql.js`, WASM) | **Consolidate into one.** Swarm DB has the richer schema (10 tables, typed entries, versioned patterns, migration tracking). Learning DB is actively written by HNSW service. Pick one engine, merge schemas. |
| **Pattern storage** | `learning-service.mjs` (SQLite short/long-term) | `agentdb_pattern-store` (RVF — never materialized) | Consolidate → use swarm DB's `patterns` table as primary. RVF is dead code. |
| **Trajectory tables** | `patterns.db` → `trajectories` (flat, no steps) | `.swarm/memory.db` → `trajectories` + `trajectory_steps` (with rewards) | Use swarm DB's richer schema (has step-level tracking + rewards). |
| **Search** | `intelligence.cjs` (trigram+Jaccard) | HNSW (learning-service.mjs) | Replace trigram with ONNX embeddings + HNSW |
| **Session persistence** | `auto-memory-hook.mjs` (JSON) | `.swarm/memory.db` → `sessions` table | Wire auto-memory to use the existing sessions table |
| **Context ranking** | PageRank (intelligence.cjs) | MMR (AgentDB adapter) | Keep PageRank for global, add MMR for query-time |
| **Consolidation** | `intelligence.consolidate()` | `agentdb_consolidate` | Wire local consolidation to call upstream |
| **KV memory** | `memory.js` (JSON — file not even created) | `memory_store` (MCP) | Drop `memory.js` if unused, or wire it to MCP |
| **HNSW metadata** | `patterns.db` → `hnsw_index` (ID mappings only) | `.swarm/memory.db` → `vector_indexes` (full HNSW config) | Use swarm DB's richer vector_indexes table |

### Systems That Should Remain Separate

| System | Reason |
|--------|--------|
| Simple KV (`memory.js`) | Ultra-fast session scratch space, no overhead |
| MEMORY.md sync | User-visible, human-editable memory |
| Pending insights log | Append-only for crash safety before consolidation |

## 4. Recommended Integration Roadmap

### Phase 1: Critical Wiring (Immediate Value)

**Goal**: Wire ONNX embeddings and core AgentDB features

| Task | Impact | Effort | Description |
|------|--------|--------|-------------|
| Wire ONNX embeddings | **Critical** | Medium | Replace trigram similarity in `intelligence.cjs` with `embeddings_generate()`. Enables true semantic search instead of character n-gram matching. |
| Wire `agentdb_feedback` to PostTask | **High** | Low | Call `agentdb_feedback(taskId, success, quality)` from post-task hook. Builds quality scores in upstream. |
| Wire trajectory hooks | **High** | Low | Call `intelligence_trajectory-start/step/end` from session and task lifecycle hooks. |
| Wire hierarchical tiers | **High** | Medium | Route storage through `agentdb_hierarchical-store/recall` instead of flat JSON. Working → Episodic → Semantic lifecycle. |

### Phase 2: Enhanced Learning (High Value)

**Goal**: Enable verdict judgment and upstream consolidation

| Task | Impact | Effort | Description |
|------|--------|--------|-------------|
| Wire verdict judgment | **High** | Medium | Compare task outcomes to historical patterns. Auto-classify success/failure. |
| Wire upstream consolidation | **Medium** | Medium | Call `agentdb_consolidate` from session-end, replacing local-only consolidation. |
| Wire semantic routing | **Medium** | Medium | Replace keyword router with `agentdb_semantic-route` for intent classification. |
| Enable batch operations | **Medium** | Low | Use `agentdb_batch` for bulk pattern inserts during consolidation. |

### Phase 3: Advanced Features (Future)

**Goal**: Full ReasoningBank pipeline, EWC++, distributed sync

| Task | Impact | Effort | Description |
|------|--------|--------|-------------|
| EWC++ integration | Medium | High | Prevent catastrophic forgetting during consolidation. Requires Fisher matrix computation. |
| LoRA distillation | Low | High | Compress patterns. Only valuable at scale (>10k patterns). |
| QUIC distributed sync | Low* | High | P2P memory sharing. Only valuable for multi-node swarms. |
| RL training (Decision Transformer) | Low* | Very High | Offline RL from trajectories. Requires substantial trajectory data first. |
| Neural training | Low | Very High | Custom model training. Premature without data volume. |

## 7. Key Risks & Gotchas

| Risk | Severity | Mitigation |
|------|----------|------------|
| **Two unsynchronized SQLite DBs** | **Critical** | `patterns.db` (better-sqlite3) and `.swarm/memory.db` (sql.js) store overlapping data with different schemas and engines. Must consolidate into one. |
| **No RVF files exist** | High | AgentDB's native RVF format is never materialized. Hybrid backend silently falls back to SQLite. This means AgentDB v3's core storage is effectively unused. |
| **Trigram search quality** | High | Wire ONNX embeddings (Phase 1). Trigram+Jaccard is a poor approximation of semantic similarity. |
| **Local HNSW is single-layer** | Medium | Use upstream AgentDB HNSW for production. Local is adequate for <2000 patterns. |
| **AgentDB v3 not in npm** | High | WM-008 patches v3 in. Must re-run `patch-all.sh` after npm cache invalidation. |
| **No background consolidation** | Medium | Currently consolidation only runs at session end. Long sessions accumulate unbounded pending insights. |
| **8+ parallel persistence stores** | High | Risk of inconsistent/divergent state across systems. Two SQLite DBs are the worst offender. |
| **Missing ControllerRegistry** | High | Upstream ADR-053 commit never published. Blocks multi-controller routing. |
| **Dead code paths** | Medium | `memory.js` references a KV file that doesn't exist. RVF code paths never produce output. |

## 8. Recommendation Summary

### Immediate (Phase 1) — 4 items, highest ROI

1. **Wire ONNX embeddings** → Transforms search quality from trigram to semantic
2. **Wire agentdb_feedback to PostTask** → Builds quality scores automatically
3. **Wire trajectory hooks** → Enables learning from execution paths
4. **Wire hierarchical tiers** → Working→Episodic→Semantic lifecycle

### Near-term (Phase 2) — 4 items, high value

5. **Wire verdict judgment** → Auto-assess pattern success/failure
6. **Wire upstream consolidation** → Better consolidation via AgentDB
7. **Wire semantic routing** → Replace keyword matching with intent classification
8. **Enable batch operations** → Faster bulk inserts during consolidation

### Defer (Phase 3) — Do not wire yet

- EWC++ (needs more pattern volume)
- LoRA distillation (needs >10k patterns)
- QUIC sync (needs multi-node usage)
- RL training (needs trajectory data)
- Neural training (premature)
- Database sharding (premature)
