## Context

ADR-0090 Tier A+B raised coverage from ~23% to ~40% with storage + daemon + controller checks. An audit of `@sparkleideas/cli@3.5.58-patch.121` then showed 213 of 239 MCP tools + 13+ CLI subcommands had zero acceptance coverage. ADR-0094 is the plan to close that gap, with the dual understanding that (a) breadth coverage is a prerequisite for behavioral coverage and (b) breadth alone is not enough — see Phases 8–10 below.

"100% coverage" is defined concretely in §Coverage Metric. It is not "every check passes" (that is a separate criterion). It is "every tuple (surface_id, depth_class) has at least one acceptance check that invokes the surface and verifies a post-condition."

## Coverage Metric

### Unit: tuple `(surface_id, depth_class)`

- **surface_id** ∈ `{mcp_tools} ∪ {cli_subcommands}` — the set of published MCP tool names + CLI subcommand names in the pinned CLI tarball. Enumerated in `config/mcp-surface-manifest.json`; regenerated by `scripts/regen-mcp-manifest.mjs`.
- **depth_class** ∈ `{happy, error, lifecycle}`:
  - `happy` — canonical inputs return canonical output shape.
  - `error` — invalid / out-of-range / malformed inputs produce a loud, diagnostic-carrying error (never a silent success).
  - `lifecycle` — for stateful tools (sessions, claims, workflows, tasks, agents, hive-mind, RVF), a multi-step sequence that exercises the transitions and verifies post-conditions at each step.

### Two-number headline

```yaml
coverage_metric:
  unit: tuple(surface_id, depth_class)
  cli_version_pinned: 3.5.58-patch.N            # auto-updated by scripts/regen-mcp-manifest.mjs
  surface_manifest: config/mcp-surface-manifest.json
  required_tuples:  <from catalog>               # |surfaces| — happy-path required for every surface
  extended_tuples:  <from catalog>               # |surfaces| × 3 depth classes
  invoked_coverage:  <from catalog>              # (pass + skip_accepted) / required_tuples
  verified_coverage: <from catalog>              # pass                    / required_tuples
  depth_coverage:    <from catalog>              # covered                 / extended_tuples
```

- **`invoked_coverage`** counts `pass` AND `skip_accepted` — it answers "do we reach the surface at all?" Target: 100%.
- **`verified_coverage`** counts `pass` only — it answers "do we prove the surface works?" Floor: ≥ 80%.
- **`skip_accepted`** is a third axis — it counts toward invocation but NOT toward verification. This is the ADR-0082-enforceable guarantee: a skip that never re-verifies rots, and the catalog's `skip_streak_days > 30 → SKIP_ROT` rule (ADR-0096) forces a review.

### Numbers are pointers, not frozen tables

ADR-0094 does **not** carry a live scoring table in its body. All coverage numbers are `<from catalog>` — the source of truth is `test-results/catalog.db` built from `test-results/accept-*/acceptance-results.json` by `scripts/catalog-rebuild.mjs`. Preflight's rot-detection fails if any number in this ADR disagrees with the catalog.

For a human-readable dashboard run: `node scripts/catalog-rebuild.mjs --show`.

## Principles

1. **One shared helper per tool category** (learned from B3/B5). Never 20 copy-pasted functions. Generic helper + thin tuple-wrapper per tool. ADR-0097 promotes this to the canonical `_mcp_invoke_tool` + `_expect_mcp_body`.
2. **Three-way bucket** (ADR-0090 Tier A2). Every check emits `pass`, `fail`, or `skip_accepted` with a narrow regex. Skips auto-flip to fail when the prereq arrives (ADR-0096 skip hygiene).
3. **No silent passes** (ADR-0082). `{success:true}` with zero side-effects is a FAIL, not a PASS. Every check must verify a POST-CONDITION (file exists, row inserted, response shape matches, state changed).
4. **Backend-appropriate verification** (learned from B5). Not everything persists to SQLite. Use file probes for RVF/redb/JSON, runtime API checks for pure-compute controllers, and state-diff checks for in-memory services.
5. **Swarm-buildable** (learned from B3/B5). Each phase should be decomposable into 3–8 parallel agents: researcher + adversarial-reviewer + builder minimum. Every swarm-generated fix MUST include an out-of-scope probe that would fail under the opposite architectural assumption (ADR-0087 addendum; see t3-2 post-mortem for precedent).
6. **Commit per phase**. Each phase produces one commit with check files + unit tests + wiring + ADR-0094-log entry.

## Phases 8–10 (behavioral coverage — P1)

Phases 1–7 proved reachability and shape-conformance. They cannot see bug classes that only manifest under cross-tool interaction, concurrency, or repetition. The t3-2 inter-process failure (→ ADR-0095), the `agentdb_experience_record` wrong-table silent success (BUG-0006), and the e2e-0059-p3 dedup failure (BUG-0007) all live in those dimensions.

### Phase 8 — Cross-tool invariants

Named invariants each exercising a mutation followed by an observation across two tools. Minimum set (6):

1. `memory_store(key, v)` ⇒ `memory_search(v)` returns the key.
2. `session_save(name)` ⇒ `session_list` shows it ⇒ `session_info(name)` returns metadata.
3. `agent_spawn(type)` ⇒ `agent_list` shows it ⇒ `agent_terminate` removes it.
4. `claims_claim(task)` ⇒ `claims_board` shows it claimed ⇒ `claims_release` clears it.
5. `workflow_create(name, steps)` ⇒ `workflow_list` shows it ⇒ `workflow_delete` removes it.
6. `config_set(key, value)` ⇒ `config_get(key)` returns value.

New check file: `lib/acceptance-phase8-invariants.sh`. Each invariant is one check function. Wall-clock budget: ≤20s total (all 6 parallelizable). Deferred to ADR-0096 design-stability.

### Phase 9 — Concurrency matrix

One race-safety check per mutation-capable surface family. Minimum set:

1. RVF writes — covered by t3-2 (→ ADR-0095).
2. Claims board — 6 agents claim simultaneously; assert exactly-one-winner.
3. Session save/restore — 2 processes save+restore same session name; assert no interleaved corruption.
4. Workflow create/execute — 4 processes start same workflow; assert one-started + queued.

New check file: `lib/acceptance-phase9-concurrency.sh`. Budget: ≤30s. **Implemented 2026-04-20** (commit `16c4b22`; 4 checks land with paired unit test `tests/unit/adr0094-p9-concurrency.test.mjs`, 13/13 pass). See ADR-0094-log.md eighth-pass entry for design notes.

### Phase 10 — Idempotency

`f(x); f(x) ≡ f(x)` for:
- `memory_store(same_key, same_value)` — no duplicate rows, returns same ID.
- `session_save(same_name)` — updates, does not create second session.
- `config_set(same_key, same_value)` — idempotent.
- `init --full` on an already-init'd dir — no-op or `--force` required.

New check file: `lib/acceptance-phase10-idempotency.sh`. Budget: ≤10s.

## Acceptance criteria

This ADR is `Implemented` when all of the following hold for 3 consecutive full-cascade runs across 3 consecutive days:

```
fail_count == 0
AND invoked_coverage == 100%
AND verified_coverage >= 80%
AND skip_streak_days_max < 30
AND preflight drift-detection passes           # manifest + catalog + run_check_bg count
AND all referenced follow-up ADRs are Implemented OR explicitly Archived
```

**Regression signal**: any `pass → fail` transition with a fingerprint matching a historical failure in `test-results/catalog.db` is reported as `REGRESSED: <check_id>`, cross-referenced to the ledger entry (if any) via `check_id`. Catalog computes the fingerprint per ADR-0096 as `sha256(check_id + first_error_line + fork_file)` truncated to 12 chars.

**Rot signal**: any number in ADR-0094 that disagrees with `node scripts/catalog-rebuild.mjs --show` fails preflight.

## Alternatives considered

### A. Test only user-facing features, not internal MCP tools
Rejected. MCP tools ARE the user-facing API for Claude Code integration. Every tool a user can `mcp exec` must have verified behavior.

### B. Rely on unit tests instead of acceptance tests
Rejected. Unit tests mock dependencies; acceptance tests exercise the real published package. Both are needed. The unit test pyramid is healthy (2850+ tests); the acceptance pyramid has gaps.

### C. 80% coverage target instead of 100%
Rejected for `invoked_coverage`; adopted for `verified_coverage`. The two-number model reconciles: every surface must be reached (100% invoked), a supermajority must be proven (≥80% verified). The gap is the `skip_accepted` bucket, forced to expire after 30 days.

### D. Keep ADR-0094 as a living tracker
Rejected (2026-04-17 hive synthesis). An ADR rewriting its own Implementation Log every 36 hours is not a decision document. The Implementation Log extracts to `docs/adr/ADR-0094-log.md`; the scoring table is regenerated from the catalog; the ADR stays as a dated snapshot.

## Maintenance Manifesto

7 rules for keeping this ADR and its sibling artifacts honest:

1. **Snapshot, not journal.** ADR-0094 body is a dated decision document. All volatile state lives in `ADR-0094-log.md`, `coverage-ledger.md`, or the catalog. Never rewrite prior ADR-0094 content; strike-through with a dated note.
2. **Numbers are pointers.** Every coverage number in this ADR is either `<from catalog>` or dated-and-struck. Preflight fails on disagreement.
3. **Fork on architecture, extend on execution.** A new design decision (own alternatives, affects protocol / architecture / size > 1 phase) gets its own ADR. Mechanical follow-through of an existing plan stays in the Implementation Log. t3-2 → ADR-0095 was the archetype.
4. **Implementation Log is append-only.** New dated H3 at the top. Corrections go in a new dated entry referencing the prior one. Prior entries never disappear.
5. **Every bug gets a ledger entry.** No inline bug tables in the ADR body. YAML entry in `coverage-ledger.md` with state + fix_commit + upstream-filed decision. Regression-matching fingerprints live in `test-results/catalog.db`, not the ledger.
6. **Swarm fixes require out-of-scope probes.** Per ADR-0087 addendum: every swarm-generated fix ships with a probe that would fail under the opposite architectural assumption (e.g., in-process guard does NOT prove inter-process correctness).
