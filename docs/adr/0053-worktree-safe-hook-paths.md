# ADR-0053: Worktree-Safe Hook Paths

## Status

Accepted

## Date

2026-03-18

## Deciders

sparkling team

## Methodology

SPARC + MADR

---

## S -- Specification

Hook commands in `.claude/settings.json` reference helper scripts using `$CLAUDE_PROJECT_DIR` as the path root. When Claude Code spawns agents in git worktrees (`.claude/worktrees/`), this environment variable may not be set or may point to the original checkout rather than the worktree root. This causes hook failures for every worktree-spawned agent.

### Affected surface

- **20 hook commands** in `generateHooksConfig()` that resolve helper paths
- **41 helper files** under `.claude/helpers/` referenced by those commands
- **1 script** (`.claude/scripts/check-patches.sh`) invoked during SessionStart
- **1 statusline** command referencing `.claude/helpers/statusline.cjs`
- **1 permissions** entry using `$CLAUDE_PROJECT_DIR` for Bash allow-listing

All paths are generated by `settings-generator.ts` in the CLI init subsystem.

### Root cause

`$CLAUDE_PROJECT_DIR` is a Claude Code environment variable injected at session start. Its behavior in worktrees is unreliable:

1. It may not be set at all in agent subprocesses
2. It may point to the parent checkout, not the worktree
3. It is not a POSIX or git standard -- it is a Claude Code implementation detail

### Why `git rev-parse --show-toplevel` is correct

`git rev-parse --show-toplevel` is a POSIX-portable git command that:

- Returns the worktree root when run inside a worktree
- Returns the repo root when run inside the main checkout
- Works regardless of CWD depth within the repo
- Is a git standard, not a tool-specific environment variable
- Helpers are git-tracked files, so they exist in every worktree

## P -- Pseudocode

```
FOR each helper function that builds hook command paths:
  REPLACE "$CLAUDE_PROJECT_DIR" with "$(git rev-parse --show-toplevel)"

Functions to patch:
  hookHandlerCmd()       -- CJS hook-handler commands
  autoMemoryCmd()        -- ESM auto-memory-hook commands
  generateStatusLineConfig() -- statusline command
  generateHooksConfig()  -- inline check-patches.sh command
  generateSettings()     -- permissions allow-list entry
```

## A -- Architecture

### Before (broken in worktrees)

```
hook command
  -> expands $CLAUDE_PROJECT_DIR
  -> may resolve to parent checkout or empty string
  -> script not found -> hook fails silently
```

### After (worktree-safe)

```
hook command
  -> runs $(git rev-parse --show-toplevel) at execution time
  -> always resolves to current repo/worktree root
  -> script found -> hook executes normally
```

### Change scope

| Function | Line | Change |
|----------|------|--------|
| `hookHandlerCmd()` | 227 | `$CLAUDE_PROJECT_DIR` -> `$(git rev-parse --show-toplevel)` |
| `autoMemoryCmd()` | 232 | same |
| `generateStatusLineConfig()` | 247 | same |
| `generateHooksConfig()` | 372 | same (check-patches.sh) |
| `generateSettings()` | 31 | same (permissions allow entry) |

### No regressions expected

- `$(...)` is standard POSIX command substitution, supported on all target shells
- `git rev-parse --show-toplevel` has been stable since git 1.7 (2010)
- The commands are shell strings executed by Claude Code's hook runner, which uses a POSIX shell
- Windows hooks already use `cmd /c` wrapping via `hookCmd()`, and `git rev-parse` works on Windows git

## R -- Review

### Alternatives considered

1. **Fix `$CLAUDE_PROJECT_DIR` in Claude Code** -- not under our control, and the variable is not part of any stable API
2. **Use relative paths** -- hooks can run from any CWD within the repo, so relative paths are equally unreliable
3. **Hardcode absolute paths at init time** -- breaks portability when the repo is moved or cloned elsewhere
4. **Use `$GIT_WORK_TREE`** -- not reliably set in all git operations

### Risk assessment

- **Low risk**: `git rev-parse --show-toplevel` is the standard way to find a repo root
- **Zero runtime cost**: the command executes in <1ms and is only called when a hook fires
- **Backward compatible**: existing installs continue to work until they re-run `claude-flow init`
