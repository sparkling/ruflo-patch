# ADR-0136: Init-generated CLAUDE.md must be AI-instruction, not user-onboarding — re-audit content principles, then add plugin/skill awareness as imperative rules

- **Status**: **Proposed (2026-05-03)**. Content-first ADR. Generator code changes are downstream of getting the content right.
- **Date**: 2026-05-03
- **Deciders**: Henrik Pettersen
- **Related**: ADR-0135 (skill+plugin enumeration audit; canonical source-of-truth for what plugin surface exists), ADR-0070 (init-generated config acceptance), ADR-0098 (anti-sprawl phrasing already in generator), ADR-0114/0115 (hive-mind council guidance already in generator)
- **Scope**: All content generated by `forks/ruflo/v3/@claude-flow/cli/src/init/claudemd-generator.ts`. **Not** the generator's wiring/templating — that's solved trivially once content is correct.

## What CLAUDE.md is for (and what it isn't)

CLAUDE.md is loaded into every Claude Code session as system instructions. The AI reads it before any user prompt and uses it to decide which tools to call, when to refuse, what conventions to follow, and what NOT to do.

**It is:** an imperative rule-set, a decision tree, an anti-pattern guide.
**It is not:** a USERGUIDE, a marketing brochure, a tutorial, a feature catalog.

Every line in CLAUDE.md must answer one of:
- "What MUST the AI do in situation X?"
- "What MUST the AI NOT do in situation X?"
- "How does the AI DECIDE between option A and option B?"
- "Where does the AI LOOK when it needs more context?"

If a line answers "what is feature X?" or "how do you install Y?" — that's USERGUIDE territory, delete it.

## AI-optimized CLAUDE.md principles (the rubric)

Drawn from Anthropic's published guidance + observed-effective patterns in this repo's own root `CLAUDE.md`:

1. **Imperative voice only.** `DO X`, `NEVER Y`, `ALWAYS Z`. No descriptive prose, no "Plugins are...", no "This system provides...".
2. **Front-load priority.** Most-violated rules first. Behavioral / safety rules at top. Reference material at bottom.
3. **Decision trees beat explanations.** Tables of "WHEN ... → USE ..." outperform paragraphs about features.
4. **Trigger-action pairs.** "When user asks for X capability missing from current toolset → run `ruflo plugins list`" is correct framing. "Plugins extend capabilities" is wrong framing.
5. **Anti-patterns explicit.** Every "DO" rule has more leverage when paired with the "DON'T" that motivates it. "DO NOT install plugins reflexively. Install only when user asks for a capability not provided by current tools" is one rule, two halves.
6. **Concrete commands over concepts.** `npm run release` (verbatim) > "the canonical release command".
7. **Reference, don't duplicate.** Catalogs (116 skills, 32 plugins, 313 MCP tools) live in USERGUIDE / `ruflo skill list` / `ruflo plugins list`. CLAUDE.md says "use the discovery command", not "here are the skills".
8. **Token-efficient.** Every word costs context budget on every turn. No filler, no padding, no rephrasing the same rule.
9. **Memory/doc pointers at the bottom.** "More context lives at `<path>`" is a one-line escape hatch from CLAUDE.md back to fuller docs when the AI hits an unfamiliar area.
10. **Terseness signals load-bearing-ness.** A bullet-rule that's 3 lines long signals "this matters". A 3-paragraph explanation signals "skip this until needed".

## Decision

**Three changes, content-first:**

### Change 1 — Add `toolSelectionRules()` (replaces the dead-code path entirely)

Decision tree for which tool primitive to reach for. Replaces the current `whenToUseWhat()` table by extending it; deprecates the dead `agentTypes()` and `memoryCommands()` functions in their current shape.

```markdown
## Tool Selection Rules

When you need a capability, choose in this order. Stop at the first match.

| You need to... | Use | Don't use |
|---|---|---|
| Run a known workflow | `Skill` tool with the skill name | Re-implementing the workflow inline |
| Coordinate parallel sub-tasks | `Agent` tool with `run_in_background: true` | `swarm_init` for one-shot work |
| Convene a council on a high-stakes decision | `hive-mind_spawn` (queen + experts + Byzantine consensus) | `Agent` fan-out (no synthesis) |
| Persist patterns/decisions across sessions | `mcp__claude-flow__memory_store` (load via ToolSearch first) | Writing to MEMORY.md from in-session work |
| Recall past decisions/patterns | `mcp__claude-flow__memory_search` | Asking the user to re-explain |
| Search the codebase | `Grep` / `Glob` / `Explore` agent | Re-reading files you've already seen |
| Run shell commands | `Bash` (single tool call, batched) | Sequential `Bash` calls when independent |

When NONE of the above match (the capability genuinely doesn't exist in the current toolset):
1. Run `ruflo skill list` — a skill may already provide it
2. Run `ruflo plugins list` — an installable plugin may provide it
3. Only after both come up empty, build the capability inline or ask the user
```

### Change 2 — Add `pluginInstallRule()` (one paragraph, imperative)

Replaces the would-be "Plugin Marketplace tutorial" with the actual AI-relevant decision: when to install a plugin vs when not to.

```markdown
## Plugin Installation Rule

DO NOT install plugins reflexively. Plugins persist after this session ends and may surface unwanted slash commands or MCP tools.

Install only when:
- The user explicitly asks for a capability provided by a known plugin (e.g., "scan for security issues" → consider `ruflo-security-audit`)
- AND `ruflo skill list` / current MCP toolset does NOT already cover it
- AND the user confirms install (plugin install changes their environment)

Discovery command: `ruflo plugins list` (also `ruflo plugins list --type integration` to filter).
Marketplace install: `/plugin install ruflo-<name>@ruflo` (after `/plugin marketplace add ruvnet/ruflo`).

After install: tell the user to run `/reload-plugins` if commands don't appear.
```

### Change 3 — Strip USERGUIDE-shaped content, replace with pointers

Remove from generated CLAUDE.md:
- Catalog enumerations ("60+ agent types: coder, reviewer, tester, ..."): replaced by 1 line `Agent types catalog: \`ruflo agent list\``
- CLI command syntax (`ruflo memory store --key X --value Y --namespace Z`): replaced by 1 line `Memory CLI: \`ruflo memory --help\``
- Setup commands (`claude mcp add ...`, `ruflo daemon start`): moved to README; the AI inherits the running env, never bootstraps it
- Marketing language ("RuFlo V3"): keep header `# Claude Code Configuration` only, no product framing

Replace each removed catalog with a "where to look" pointer:

```markdown
## Reference Pointers (when you need more than this file says)

- Tool catalog: ToolSearch with a relevant query
- Skill catalog: `ruflo skill list`
- Plugin catalog: `ruflo plugins list`
- Agent type catalog: `ruflo agent list`
- ADR decisions: `docs/adr/`
- Project memory (cross-session lessons): `~/.claude/projects/<project>/memory/MEMORY.md`
- USERGUIDE (full feature surface): https://github.com/ruvnet/claude-flow/blob/main/docs/USERGUIDE.md
```

## Acceptance criteria (content-shape, not code)

A correct CLAUDE.md from `ruflo init --template standard` MUST:

1. Contain ZERO occurrences of the strings `marketplace`, `plugin install`, `npm install` outside of `## Plugin Installation Rule` (the only place plugin install is mentioned, and only in the context of WHEN-to-install rule)
2. Contain ZERO product-marketing language (no "comprehensive", no "powerful", no "extensive", no "x faster than y" benchmark claims)
3. Contain ZERO catalog enumerations longer than 5 items inline; longer catalogs go behind a discovery command pointer
4. Contain at least 5 imperative rules per major section (`DO X`, `NEVER Y`, `ALWAYS Z`)
5. Contain at least 1 decision-tree table (When X → Use Y format)
6. Be under **120 lines for `standard`**, under **180 lines for `full`** (current standard is ~90; the proposed additions stay within budget by stripping USERGUIDE-shaped content elsewhere)
7. Pass a "would the AI know what to do?" review on a sample task: "user asks to scan code for security issues" — CLAUDE.md should let the AI navigate to the right tool (security-auditor agent / ruflo-security-audit plugin / ruflo security scan CLI) without re-reading USERGUIDE.

## What this ADR is NOT proposing

- **Not proposing skill enumeration.** The 116-skill surface from ADR-0135 stays in USERGUIDE. CLAUDE.md just says "see `ruflo skill list`".
- **Not proposing plugin catalog.** The 8 marketplace + 14 npm + 6 WASM + 19 IPFS plugins stay in USERGUIDE. CLAUDE.md says "see `ruflo plugins list`".
- **Not proposing setup-flow content.** Setup is README's job; the AI doesn't bootstrap.
- **Not proposing template wiring.** That's a 5-line PR after the content is approved.

## Trade-offs

- **Loses the discoverability hook.** A user reading their generated CLAUDE.md might never learn the marketplace exists. Counter: the marketplace is README/USERGUIDE territory; CLAUDE.md is for the AI mid-session, when the user has already arrived. The right discoverability surface is `ruflo init` printing post-install hints + README.
- **Strips user-friendly setup commands from CLAUDE.md.** Counter: those commands are run before CLAUDE.md is read; once the AI sees CLAUDE.md, the env is already up. Setup belongs upstream of init's CLAUDE.md.
- **Goes against the impulse to make CLAUDE.md "complete".** Counter: the goal is highest-leverage instructions per token, not completeness. Pointers > catalogs.

## Why this beats the previous draft of this ADR

Earlier draft (this same file, prior version) proposed `pluginDiscovery()` + `skillDiscovery()` sections that read like USERGUIDE excerpts — listing 8 marketplace plugins by name, listing 7 skill categories with 25+ slash-command examples, showing install commands inline. That's tutorial. This rewrite re-frames the same gap as a **decision rule** (when to install) plus a **discovery pointer** (run the catalog command on demand). Same gap closed, ~150 lines saved, AI gets a sharper instruction.

## Open questions (deferred)

1. Should `agentOrchestration()`'s "DEFAULT: Agent tool" framing be made even sharper? Currently 8 lines explain when to use Agent vs hive-mind_spawn. Could be a 4-row table.
2. Should there be a `postInitHints()` that runs in the init CLI itself (printing to terminal, not CLAUDE.md), telling the user "to add plugins, run X" — replacing the role the previous draft tried to put into CLAUDE.md?
3. Should the security and performance templates strip the protocol blocks (`## Security Protocol`, `## Performance Optimization Protocol`) further? They lean toward USERGUIDE-shape too. Out of scope for this ADR — separate audit.

## Consequences

- **Positive**: Generated CLAUDE.md becomes load-bearing on every session start. Plugin/skill awareness lands as decision rules the AI applies, not as catalog the AI ignores.
- **Positive**: Cuts ~50 tokens off every Claude Code session that uses an init'd ruflo project — the marketing/setup/catalog content was paid for on every turn.
- **Positive**: Establishes a re-usable rubric (the 10 principles) for future generator audits. Other generators (settings.json, hooks.json) can be scored the same way.
- **Negative**: Diverges from upstream further. Acceptable.
- **Risk**: The acceptance criteria are content-shape, not behavioral. Possible to satisfy them mechanically and still produce bad CLAUDE.md. Mitigation: criterion 7 (the "would the AI know what to do?" sample-task review) is the actual gate.
